# Авто-Рилс — описание проекта

Инструмент автоматически режет длинное горизонтальное видео на короткие вертикальные рилсы (9:16) **без ручного монтажа**. Облако выбирает моменты и отдаёт текстовый план нарезки; локальное железо рендерит по плану.

---

## Цель и рамки (MVP)

- **Что делает:** длинное видео (~60 мин) → набор готовых вертикальных рилсов с выжженными субтитрами.
- **Для кого:** персональное использование, расширение на узкий круг — позже.
- **Тип контента MVP:** talking-head, один человек, статичен и по центру кадра. Это упрощение снимает самую дорогую часть (трекинг говорящего) — её в MVP нет.
- **Режим:** полностью автоматический («без рук»). Ручная доводка в редакторе — опциональная ветка, не основной поток.

---

## Ключевой архитектурный принцип: граница ОБЛАКО / ЛОКАЛКА

Единственный несущий концепт проекта. Конвейер физически разделён на два тира, и это разделение объясняет всё остальное (где живут лимиты, что передаётся между этапами, что бесплатно).

| | ОБЛАКО (API) | ЛОКАЛКА (железо) |
|---|---|---|
| Делает | транскрипция, выбор моментов | нарезка, кроп, субтитры, рендер |
| Инструменты | Groq (Whisper + Qwen) | ffmpeg |
| Стоимость | бесплатный тариф, упирается в лимиты | бесплатно по определению, лимитов нет |
| Узкое место | **TPM на LLM-выборке** | скорость рендера на 8 ГБ GPU |
| На выходе | **JSON-манифест** (план), не видео | mp4-файлы рилсов |

**Видео между тирами не передаётся.** Облако работает только с текстом (транскрипт → план) и отдаёт лёгкий манифест. Исходник всё время лежит локально, где и рендерится. Это и делает «бесплатные API» реалистичными: тяжёлое не уходит наружу.

**Приём исходника — файл или ссылка.** Кроме локального файла, на вход принимается ссылка (YouTube / VK / RuTube). `yt-dlp` на рендер-машине качает исходник **сразу туда, где он будет рендериться** — идеально ложится в правило «исходник живёт на рендер-машине». Ссылка (КБ) передаётся между машинами свободно, само видео не движется.

---

## Поток файлов и приём исходника

Где физически лежит видео и как оно попадает в конвейер. Несущее правило прежнее: **видео между тирами не передаётся**, в облако уходит только звук/текст.

### Где живёт исходник (три варианта; для MVP — вариант 1)

- **Вариант 1 (MVP, рекомендован).** Видео лежит на ПК-рендере (системник). R0 (`extract_audio` → `transcribe`) гонится **там же**. В облако уходит только звук, не видео. Видео не двигается вообще. Ноут M1 — только разработка кода, исходники там не рендерятся.
- **Вариант 2.** Видео на ноуте, рендер на системнике → исходник **один раз** переносится ноут→системник (Syncthing / Tailscale), дальше между машинами ездит только манифест.
- **Вариант 3 (M2).** Видео на видеохостинге/в облаке → системник сам качает по ссылке, между машинами едет только ссылка + манифест.

### Слой приёма (взаимозаменяемые доставщики)

- Программа **всегда** читает локальный путь `inputs/<файл>`. Способ доставки — отдельный слой, который кладёт файл на этот путь. Ядро от способа доставки **не зависит**.
- Доставщики (взаимозаменяемы): `yt-dlp` (видеохостинг), HTTP-загрузка (прямая ссылка на облако), Tailscale (`rsync`/`scp` с другой машины), Syncthing (синк-папка = `inputs/`), ручное копирование. Все кладут файл в `inputs/`, дальше — единый поток.
- Схема: `[доставщик] → inputs/<файл> → extract_audio → … → render`.

### source-привязка в манифесте

- Поле `source` = **имя файла + sha256 содержимого** (ключ привязки).
- Рендерер ищет файл по **хэшу** в `inputs/`, а не только по имени → манифест не применится к чужому файлу-тёзке (защита от перепутанных исходников).

## Конвейер по стадиям

Дешёвый текстовый этап стоит первым и работает стоп-чеком: если выборка моментов плоха, рендерить нечего.

### R0 — Выбор моментов *(облако, единственный реальный чекпоинт)*
Транскрипция (Whisper, word-level таймкоды) → LLM (Qwen) ранжирует сегменты по рамке виральности (hook, эмоц. пик, цитируемость, вопрос-ответ) и возвращает кандидатов `{start, end, score, hook, title, description, reason}`. Длинный транскрипт → чанкинг + дедупликация пересечений (>50% overlap → выше по скору).

**Правило самодостаточности (из практики Reels Boss — клипы «до минуты»):** каждый сегмент обязан укладываться в `max_duration` и быть законченным по смыслу. Сильный фрагмент длиннее лимита — LLM ужимает до пуанта или разбивает; код-валидатор ставит флаг `too_long` и режет/отбраковывает то, что не уложилось. Длина — конфиг, не хардкод:

| Пресет | min | max | Платформа |
|---|---|---|---|
| `shorts` (дефолт) | 15 | 59 | YouTube Shorts, универсал |
| `reels` | 15 | 90 | Instagram Reels |
| `tiktok` | 15 | 180 | TikTok |

**Заголовок + описание на клип:** `title` (виральный) и `description` генерируются тем же вызовом R0 — ноль новой инфраструктуры, просто поля в кандидате. Уходят в манифест и дальше в SMM-постер как готовый текст публикации.

**Стоп-чек:** моменты осмысленные и самодостаточные в пределах лимита? Если нет — дальше не идём.

### R1 — Детерминированная нарезка *(локалка)*
ffmpeg режет по границам слов → фикс-кроп 9:16 → сырой вертикальный mp4. Без украшений. Доказывает, что из таймкодов получается валидный клип.

### R2 — Оркестрация + review + выдача
FastAPI-сервис + очередь. Review-экран (переиспользуется паттерн из Meeting→Tasks: загрузка, inline-правка таймкодов, per-clip approval). Approve → готовые mp4 уходят в SMM кросс-постер. Петля замкнута: видео → опубликованный рилс.

### R3 — Субтитры *(локалка, на практике must)*
Word-level → **ASS-шаблон** (один настроенный `Style:` блок: шрифт, позиция в зоне 1080×1920, поп-ап по словам) → ffmpeg выжигает. Смотрят без звука — поэтому фактически часть MVP.

### R4 — Границы по смыслу *(полировка)*
PySceneDetect + snap начала/конца к паузам и словам; вытягивание hook в первые секунды. Про «ощущается смонтированным», не про работоспособность — поэтому в конце.

## Манифест — формат «проекта»

Граница между тирами. Лёгкий JSON, на каждый рилс:

```json
{
  "source": "lecture_2026.mp4",
  "duration_preset": "shorts",
  "reels": [
    {
      "id": "r01",
      "start": 124.3,
      "end": 168.9,
      "hook": "первая фраза-зацепка",
      "title": "виральный заголовок клипа",
      "description": "готовый текст публикации для соцсетей",
      "score": 87,
      "crop": { "x": 980, "y": 220, "w": 1010, "h": 1795 },
      "subtitles": [
        { "word": "привет", "t0": 124.3, "t1": 124.6 }
      ]
    }
  ]
}
```

- **`source`-привязка:** имя файла + sha256 содержимого — рендерер ищет исходник по хэшу в `inputs/` (см. «Поток файлов и приём исходника»). В схеме это TODO к шагу 8/CLI.
- **Основной выход:** свой JSON → рендерит ffmpeg-скрипт (поток без рук).
- **Опциональный выход:** fcpxml → открывается в **DaVinci Resolve** для ручной доводки тех 10% клипов, где хочется цвет/тонкую анимацию титров. CapCut не закладываем — формат закрытый и хрупкий.

Один пайплайн — два выхода из одного манифеста.

## Визуальная часть (под talking-head, статичная камера)

- **Кроп — фиксированный прямоугольник `{x, y, w, h}` в пикселях исходника**, не геометрический центр. Камера и человек статичны → рамка **калибруется один раз на сетап** (профиль) и покрывает всё видео и все клипы. `ffmpeg -vf "crop=w:h:x:y, scale=1080:1920"`. После разовой калибровки — полностью без рук.
  - Калибровка учитывает обе оси: по горизонтали — колонка с человеком (может быть смещена от центра кадра); по вертикали — окно от макушки до нужной нижней границы, передний план/лишнее отрезается.
  - Если исходник ультраширокий (шире 16:9) — рамка просто меньше полной высоты, логика та же.
- **Трекинг / active-speaker detection:** не нужны. Обходятся статикой кадра + разовой калибровкой.
- **Субтитры:** ASS-шаблон (стиль ниже), настраивается один раз, рендерится ко всему потоку автоматически.

### Профиль сетапа (калибруется один раз)
```json
{
  "setup_id": "tearoom_main",
  "crop": { "x": 980, "y": 220, "w": 1010, "h": 1795 },
  "scale": [1080, 1920]
}
```

### ASS-стиль субтитров (с рабочего примера)
Белый, жирный, CAPS, по центру, нижняя треть; тонкий контур + лёгкая тень для читаемости на любом фоне; **без подложки-бокса**. Поп-ап по словам через `\k`, 2–4 слова в строке.
```
Style: Pop,Montserrat,72,&H00FFFFFF,&H00FFFFFF,&H00000000,&H64000000,-1,0,0,0,100,100,0,0,1,3,2,2,40,40,300,1
```
Координаты в пространстве кропнутого кадра 1080×1920; `MarginV ≈ 300` сажает текст в нижнюю треть, выше зоны UI платформы.

## Откуда что берём (не форк, а сборка)

Проект сужен (только бесплатные API, локальный ffmpeg-рендер через манифест, talking-head по центру без трекинга), поэтому форкать один репозиторий целиком неоптимально — у всех зашит рефрейминг/трекинг, которого тут нет, и рендер идёт in-process, а не через манифест. Берём по частям:

| Источник | Что забираем | Что игнорируем |
|---|---|---|
| **SamurAIGPT/AI-Youtube-Shorts-Generator** | Логика R0: рамка виральности, чанкинг длинных видео, дедуп пересечений, `--mode local` (faster-whisper + ffmpeg). Ближе всего к нашей архитектуре. | — |
| **OpenShorts** (mutonby) | R3: ASS-субтитры с word-level стилизацией (поп-ап по словам), готовый стиль. | Весь YOLOv8+MediaPipe трекинг, Next.js SaaS-обвязка |
| **autoshorts** (divyaprakash0426) | Опционально: NVENC-рендер трюки для скорости на слабой карте. | Action-скоринг (не по транскрипту), gameplay-специфика |

**Что не берём ни у кого** (наш собственный слой):
- Манифест-слой (облако → JSON → отдельный рендерер). У всех рендер in-process — этого паттерна нет ни у кого.
- Reframing/tracking — заменён одним статичным crop (несколько строк ffmpeg).

Итог: сборка легче любого исходника, потому что выкинута самая жирная часть — трекинг.

## Жёсткие правила и принципы

Перенесены из Meeting→Tasks один-в-один:

- **Determinism-first.** LLM только ранжирует и предлагает. Финальные границы реза выставляет детерминированный код (snap к словам/паузам/сценам), не «уверенность» модели. Скоры — подсказка для сортировки, не источник истины.
- **Пустой массив — валидный результат.** «Хороших моментов не найдено» — ожидаемый исход для скучного видео, не ошибка.
- **Two-phase.** Phase 1 (R0→R1) гонит автоматически до `awaiting_review`. Phase 2 (финальный рендер + выдача) — только по явному approve.
- **Token-aware на R0.** Сжатие транскрипта (sentence-level для LLM, word-level только для реза) + чанки под 6K TPM + троттлинг по `x-ratelimit` заголовкам + prompt cache на рубрику скоринга.
- **Чек-флаги детерминированы.** «Клип слишком короткий / режется посреди фразы / нет hook» ставит код, не модель.

## На горизонте (вне MVP)

- Рефрейминг с трекингом говорящего (для контента, где человек двигается / несколько спикеров).
- Разовый детект лица для вертикальной посадки (если съёмка непостоянна).
- Масштабирование на 3–5 пользователей.
- Локальный Qwen (когда появится GPU ≥16 ГБ) → снимает лимиты R0 полностью.
