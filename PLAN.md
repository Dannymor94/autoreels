# PLAN.md — Авто-Рилс: план реализации

Инкрементальный, каждый milestone даёт реальный выход до начала следующего. Стоп-чеки явные.

Стадии конвейера (R0–R4) — из `PROJECT_GUIDE.md`. Здесь — **порядок сборки** (milestones M0–M2 → v1.0), который режет конвейер вертикальными слайсами.

---

## M0 — Один клип end-to-end (вертикальный слайс)

**Цель:** локальный файл → один валидный вертикальный рилс с субтитрами. Доказать жизнеспособность всей цепочки. CLI, без UI, без review (auto-approve), профиль кропа захардкожен в конфиг.

Порядок (критический путь из анализа: рубрика → аудио → калибровка → рендер):

1. **Каркас + модели.** Репо по `PROJECT_STRUCTURE.md`, Pydantic-схема манифеста (`core/models.py`), конфиг-загрузка YAML, TDD-scaffold.
2. **Извлечение аудио.** `ffmpeg -vn` → wav/m4a. Тест: длительность совпадает.
3. **Транскрипция.** faster-whisper локально (word-level). Кэш по хэшу аудио (идемпотентность). Тест на коротком клипе.
4. **Сжатие транскрипта.** word-level → sentence-level + таймкоды (`cloud/compress.py`). Тест: токены ↓ ~2×, таймкоды сохранены.
5. **R0 — выбор (ядро).** Провайдер-слой (Groq) + промпт по `R0_SPEC.md` + парсинг строгого JSON + валидаторы (длина, score) + дедуп. Чанкинг пока можно опустить (взять короткое видео ≤10 мин).
   - ⛔ **СТОП-ЧЕК:** прогнать на реальном транскрипте, глазами проверить выборку. Плохо → чинить рубрику, не идти дальше.
6. **R1 — нарезка.** ffmpeg cut по таймкодам + статичный кроп из профиля → mp4. Тест: разрешение 1080×1920, длительность в пресете.
7. **R3 — субтитры.** word-level → ASS (стиль из `PROJECT_GUIDE.md`, группировка 2–4 слова) → burn-in. 
8. **Склейка в CLI.** `python -m autoreels run video.mp4 --setup tearoom_main` → папка с mp4 + `manifest.json`.

**Выход M0:** из реального видео получены 1–N клипов, которые не стыдно опубликовать. Если да — фундамент доказан.

## M1 — Полный R0 + review + заголовки

**Цель:** надёжный R0 на часовых видео + человек в петле + готовый текст публикации.

1. **Чанкинг длинного транскрипта** (overlap + дедуп поверх чанков) + троттлинг по заголовкам + prompt-cache. Тест на 46-мин транскрипте под лимиты Groq free.
2. **OpenRouter failover** в провайдер-слое.
3. **Кликбейт `title` + `description`** в R0 (рамка из `R0_SPEC.md` §4) + few-shot примеры с реального контента.
4. **R2 — оркестрация + review-UI.** FastAPI + очередь + статусная модель (`awaiting_review`). React-экран: превью клипа, правка in/out, правка заголовка, approve/reject. (Скилл `frontend-design`.)
5. **R4 — snap по сценам** (PySceneDetect) + притяжка границ к паузам/словам. Полировка качества реза.

**Выход M1:** грузишь часовое видео → ревьюишь карточки клипов в браузере → approve → готовые mp4 + тексты.

## M2 — Приём по ссылке + стык с экосистемой

1. **yt-dlp приём** (YouTube/VK/RuTube) — качает исходник на рендер-машину. Вход: файл ИЛИ ссылка.
2. **Перенос между машинами** — Tailscale (рендер-машина как сервер, рабочий комп в браузере) ИЛИ Syncthing (inbox/outbox). Зафиксировать выбор.
3. **Контракт с SMM-постером** — манифест (`title`/`description`/хэштеги + mp4) → формат, который ждёт кросс-постер. Это и есть «привязка проекта».

**Выход M2:** вставил ссылку → клипы ушли в SMM-инструмент на публикацию. Петля экосистемы замкнута.

## Зависимости и риск

- Единственный реальный риск — **качество R0** (M0 шаг 5). Всё остальное — предсказуемая инженерия.
- M0 шаги 2–4, 6–7 независимы от LLM, тестируются изолированно (TDD на детерминированном слое).
- Чанкинг (M1.1) — то, без чего часовое видео не проходит бесплатный Groq. До него работать на коротких видео.
