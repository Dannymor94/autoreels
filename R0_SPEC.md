# R0 — Выбор моментов: детальная спецификация

Ядро Авто-Рилс и единственное место реального риска. Облачный этап: транскрипт → кандидаты-клипы с заголовками. Всё остальное — детерминированная инженерия вокруг этого.

Принцип, который держит всю стадию: **LLM предлагает и ранжирует, код решает и валидирует.** Скоры и границы от модели — черновик; финал ставит детерминированный слой.

---

## 0. Поток данных R0

```
аудио (ffmpeg -vn)
  → транскрипция (Whisper, word-level)
    → нормализация в sentence-level + таймкоды        [код]
      → сжатие транскрипта под бюджет токенов          [код]
        → чанкинг (если длинный)                       [код]
          → LLM-вызов на каждый чанк                   [LLM]
            → парсинг + валидация кандидатов           [код]
              → дедуп пересечений между чанками        [код]
                → snap границ к словам/паузам          [код]
                  → ранжирование, отбор топ-N          [код]
                    → манифест (awaiting_review)
```

LLM трогается ровно в одной точке. Всё до и после — детерминированный код.

---

## 1. Вход в LLM: сжатый формат транскрипта

Word-level нужен **коду** (для snap границ при нарезке), но **не LLM** — модели хватает предложений с таймкодами. Отправка word-level в LLM сожгла бы бюджет токенов впустую.

Формат, который уходит в модель (одна строка = одно предложение):

```
[0124.3-0131.0] Самый важный момент в работе с травмой — это не сама травма.
[0131.0-0138.4] За ней всегда скрыт ресурс, который человек не замечает.
[0138.4-0145.1] И вот тут начинается самое интересное.
```

Правила сжатия (код, до отправки):
- предложения склеиваются из word-таймкодов по границам пауз/пунктуации;
- таймкоды — только начало-конец предложения, не каждое слово;
- филлеры-страховки («эээ», «ну вот», повторы) схлопываются;
- числовой индекс предложения **не нужен** — модель адресует сегменты таймкодами, и это же ключ для обратной привязки к word-level на этапе нарезки.

Экономия: ~2× против сырого word-level JSON. Это разница между «влезает в 6K TPM чанком» и «не влезает».

## 2. Рамка виральности (рубрика скоринга)

Модель оценивает каждый потенциальный сегмент по сигналам. Рубрика идёт в **system-промпт** (стабилен → кэшируется, не ест бюджет):

**Сильные сигналы (повышают score):**
- **Hook в первых 3 сек** — сегмент открывается зацепкой, а не разгоном. Без хука клип мёртв.
- **Самодостаточность** — понятно без контекста остального видео.
- **Эмоциональный пик** — удивление, конфликт, инсайт, переворот ожидания.
- **Цитируемость** — есть фраза, которую хочется пересказать.
- **Вопрос → ответ** — закрытая микро-арка внутри клипа.
- **Контринтуитивность** — «на самом деле всё наоборот».

**Анти-сигналы (понижают / отбраковывают):**
- обрыв на полуслове, ссылка на «как я говорил раньше», незаконченная мысль;
- организационное («сделаем перерыв», «включите звук»);
- длинный разгон без пуанта.

**Калибровка score (0–100):** ≥80 — публиковать смело; 60–79 — годится; <60 — слабо, в норме не берём. Порог отбора — конфиг (`min_score`, дефолт 65).

## 3. Правило длины и самодостаточности

Из практики (клипы «до минуты»): каждый кандидат **обязан** укладываться в `max_duration` пресета и быть законченным.

- Если сильный момент длиннее лимита → модель инструктируется **ужать до пуанта** (сдвинуть start ближе к развязке) или **разбить на два** самостоятельных клипа.
- Код-валидатор после LLM: `duration > max` → флаг `too_long` → либо обрезка по правилу, либо отбраковка.
- `duration < min` (дефолт 15 сек) → флаг `too_short` → отбраковка (слишком коротко для смысла).

Пресеты — из PROJECT_GUIDE (`shorts` 15–59 дефолт / `reels` / `tiktok`).

## 4. Кликбейт-формула заголовка

`title` генерируется тем же вызовом, но **не свободно** — по паттерну, иначе Qwen выдаёт пресный заголовок. Рамка (в промпт):

Паттерны (выбрать подходящий):
- **Скрытое** — «За X скрыт Y», «X на самом деле про Y»
- **Переворот** — «почему [группа] на самом деле [контринтуитив]»
- **Curiosity gap** — обещание ответа без его раскрытия

Обязательные элементы:
- **CAPS на 2–3 эмоциональных словах** (не весь заголовок);
- **1 эмодзи-акцент** по смыслу;
- **обрыв «…»** в конце — провоцирует тап.

Эталон (с рабочего клипа): `ЗА ТРАВМОЙ скрыт ДАР: почему «беглецы» на самом деле СЕРДЕЧНЫЕ 🫀…`

`description` (1–2 фразы): пересказ крючка без раскрытия развязки + эмодзи + 3–5 хэштегов.
Эталон: `Самый неожиданный поворот: за травмой может прятаться ваш ДАР. И если вы сейчас в боли — это не «сломался», это нервная система 🫀 #психотерапия #самооценка #травма #психология`

## 5. Контракт выхода LLM (строгий JSON)

Модель возвращает **только** JSON, без преамбулы и markdown. Парсинг — с fallback на повторный вызов при невалидном JSON (как provider-failover в Meeting→Tasks).

```json
{
  "segments": [
    {
      "start": 124.3,
      "end": 168.9,
      "score": 87,
      "hook": "За травмой всегда скрыт ресурс",
      "title": "ЗА ТРАВМОЙ скрыт ДАР: почему «беглецы» СЕРДЕЧНЫЕ 🫀…",
      "description": "Самый неожиданный поворот… 🫀 #психотерапия #травма #психология",
      "reason": "контринтуитивный тезис + закрытая арка, цитируемо",
      "topic": "травма как ресурс"
    }
  ]
}
```

Пустой `segments: []` — **валидный результат**. «В этом чанке хороших моментов нет» — норма, не ошибка.

## 6. Чанкинг длинного транскрипта

60-мин видео ≈ 13K токенов сжатого транскрипта > 6K TPM Groq. Значит:

- **Размер чанка** — под бюджет: `chunk_tokens ≤ TPM − reserve` (reserve на output + system). Для Groq free дефолт ~3500 input-токенов на чанк.
- **Overlap** — соседние чанки перекрываются на ~60 сек, чтобы момент на стыке не потерялся.
- **Адресация по таймкодам** — сегменты из разных чанков сравнимы по абсолютному времени → дедуп работает поверх всех чанков.

---

## 7. Дедупликация и финальный отбор (код, после всех LLM-вызовов)

1. **Сбор** всех `segments` со всех чанков в один пул.
2. **Дедуп пересечений:** если два сегмента overlap > 50% → оставить с большим `score`, второй выкинуть.
3. **Snap границ** к word-level: start/end притянуть к ближайшей границе слова и паузе (не резать посреди слова). Здесь и нужен сохранённый word-level из п.1.
4. **Фильтр** по `min_score`, `min_duration`, флагам.
5. **Ранжирование** по score, отбор топ-N (или все прошедшие порог — конфиг `max_reels`, `null` = все).
6. Запись в манифест, статус `awaiting_review`.

---

## 8. Token-aware провайдер-слой

Переиспользуется абстракция из Meeting→Tasks (`providers.py`), плюс специфика R0:

- **Round-robin Groq → OpenRouter** при исчерпании лимита/ошибке.
- **Троттлинг по заголовкам:** читать `x-ratelimit-remaining-tokens` / `-requests`, спать до `x-ratelimit-reset` вместо слепых ретраев.
- **Prompt cache на system-промпт** (рубрика + кликбейт-формула стабильны) — кэш-токены не считаются в лимит.
- **Бэкофф** на 429: экспоненциальный, с уважением `retry-after`.
- **Бюджет на видео:** логировать суммарные токены/запросы на один прогон — для контроля дневного потолка (RPD).

---

## 9. Идемпотентность

Ловушка, уже пойманная в Meeting→Tasks (баг размножения задач) — здесь та же:

- ключ прогона = хэш(`source` + `duration_preset` + версия рубрики);
- транскрипт кэшируется по хэшу аудио → повторный запуск не перетранскрибирует и не дёргает Whisper заново;
- сегменты кэшируются по ключу прогона → повторный запуск не плодит дубли клипов;
- перегенерация — только при явном изменении входа или флаге `--force`.

## 10. Конфиг R0 (всё, что вынесено из кода)

```yaml
duration_preset: shorts        # shorts | reels | tiktok
min_score: 65
max_reels: null                # null = все прошедшие порог
chunk_tokens: 3500
chunk_overlap_sec: 60
dedup_overlap_threshold: 0.5
title_style: clickbait_ru      # рамка из п.4
language: ru                   # вывод
prompt_language: en            # system-промпт на английском (сильнее grounding)
```

`prompt_language: en` — перенос валидированного в Meeting→Tasks паттерна: английский system-промпт + русский вывод даёт более крепкий grounding на неякорной русской речи.
